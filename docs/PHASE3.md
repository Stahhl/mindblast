# Phase 3 Specification: AI-Assisted Distractor Selection

## Naming
- Project name: `Mindblast`.
- User-facing app name: `Mindblast`.
- Backend generator service name: `quiz-forge`.

## Objective
Introduce a low-risk AI workflow in `quiz-forge` that improves MCQ distractor quality while keeping reliability, budget, and compliance constraints strict.

Phase 3 starts with `history_mcq_4` only.

## Product Intent
- Improve question quality by selecting more plausible distractors.
- Build an AI orchestration pattern that can be extended to future tasks.
- Keep deterministic fallback behavior so daily generation remains reliable.

## Scope (Phase 3)
- Add an AI task for `history_mcq_4`: `rerank_distractors`.
- Keep 4/4 answer options as real sourced facts in production.
- Support two provider modes:
  - on-device model provider
  - external API provider
- Add strict spending and usage limits with fail-closed + fallback behavior.
- Add shadow-mode operation to compare AI choices without changing published output.

## Out of Scope (Phase 3)
- Synthetic/fabricated answer options in production output.
- Expanding AI to all quiz types.
- Introducing user-facing explanation text generated by AI.
- Paid high-throughput inference infrastructure.

## Non-Negotiable Guardrails
1. Do not fabricate sources.
   - `source.events_used` must only contain real, attributable source events.
2. Synthetic content is blocked for production output in this phase.
3. If synthetic content is enabled in a future phase:
   - it must be explicitly labeled in machine-readable metadata and UI,
   - it must never claim a real citation URL,
   - it must never be mixed into source-attribution lists as if sourced facts.
4. If AI call fails, times out, or breaches budget, fallback to deterministic selection.

## Workflow (history_mcq_4)
1. Build deterministic candidate pool:
   - choose 1 correct fact,
   - choose N distractor candidates (`N > 3`) from real source events.
2. Call AI `rerank_distractors` task with strict constraints.
3. Validate AI output:
   - only candidate IDs from provided pool,
   - no duplicates,
   - distinct-year rules remain satisfied,
   - output count is exactly 3.
4. Compose final options:
   - correct + top 3 validated distractors.
5. If any validation fails:
   - discard AI output,
   - use deterministic distractor selector.
6. Continue normal schema validation and write flow.

## AI Runtime Modes
- `AI_MODE=off`
  - AI path disabled, deterministic only.
- `AI_MODE=shadow`
  - AI called, result logged/compared, deterministic output still published.
- `AI_MODE=on`
  - validated AI ranking is used for published distractors.

Default for initial rollout: `off` in production, `shadow` in staging.

## Provider Architecture Contract
Define task/provider interfaces under `scripts/quiz_forge/ai/`:
- `providers/base.py`: provider interface and shared request/response types.
- `providers/local.py`: on-device implementation.
- `providers/external.py`: remote API implementation.
- `tasks/rerank_distractors.py`: task-level prompt/constraints/validation.
- `budget.py`: spending and usage guardrails.
- `orchestrator.py`: mode handling (`off`, `shadow`, `on`) + fallback control.

Provider response contract (minimum):
- ordered distractor candidate IDs,
- model/provider identifier,
- token usage/cost fields when available,
- timestamp for audit logging.

## Budget and Safety Controls
Environment-controlled hard limits:
- `AI_MAX_DAILY_USD`
- `AI_MAX_CALLS_PER_RUN`
- `AI_MAX_INPUT_TOKENS`
- `AI_MAX_OUTPUT_TOKENS`
- `AI_TIMEOUT_MS`

Rules:
- If any limit is exceeded, do not publish AI-influenced output.
- Log budget/fallback reason and continue deterministic path.
- Never block daily quiz generation solely due to AI unavailability.

## Observability
For each run, record concise AI telemetry in logs:
- mode, provider, task, latency, token/cost estimate, fallback reason.

Optional (recommended) artifact in CI:
- `artifacts/ai_shadow_report_<date>.json` with AI vs deterministic comparison.

## Data Contract Impact
Phase 3 should preserve current quiz payload schema compatibility.
- Existing consumers must continue to work without requiring AI-specific fields.
- Any new AI metadata in payloads must be additive and optional.

If synthetic content is introduced in a later phase, contract must be updated first in docs before implementation.

## Rollout Plan
1. Scaffold AI module interfaces and no-op provider.
2. Implement deterministic + AI rerank comparison in `shadow` mode.
3. Add budget guard + fallback tests.
4. Validate quality deltas on staging.
5. Enable `AI_MODE=on` only after acceptance criteria are met.

## Acceptance Criteria
- Daily generation remains idempotent and stable with AI enabled or disabled.
- In `off` mode, output is unchanged from deterministic baseline.
- In `shadow` mode, output remains deterministic while comparison telemetry is produced.
- In `on` mode, AI-ranked distractors are used only when fully valid and within budget.
- No synthetic options are published in Phase 3.
- No fabricated/incorrect source attribution appears in output.
